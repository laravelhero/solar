# Apex Solar — Premium USA Solar Energy Website

A production-ready, SEO-first marketing site for a U.S. solar energy company, built with **Astro**, **Tailwind CSS v4** and **Alpine.js**. Static-generated, mobile-first, WCAG-aware, and fully data-driven for easy CMS migration later.

- **No React / Vue / Svelte** — just Astro + a sprinkle of Alpine for interactivity
- **80 statically-generated pages** from ~10 data files + a content collection
- **Premium 2026 design system** — glassmorphism, soft shadows, gradient accents
- Brand palette: **Solar Orange** (primary) · **Energy Blue** (secondary) · **Green Energy** (accent)

```sh
npm install
npm run dev      # http://localhost:4321
npm run build    # static output → ./dist
npm run preview  # preview the production build
```

## Architecture at a glance

```
src/
├── content.config.ts         # blog content collection schema
├── content/blog/*.md         # 10 SEO blog articles (frontmatter + body)
├── data/                     # ← single source of truth (edit content here)
│   ├── company.ts            # NAP, ratings, certifications
│   ├── navigation.ts         # main nav + mega-menu + footer (built from services)
│   ├── services.ts           # 18 services → mega-menu + /services/[slug]
│   ├── projects.ts           # 20 projects → /projects/[slug] + case studies
│   ├── locations.ts          # 8 states → /locations/[slug] (local SEO)
│   ├── testimonials.ts · faqs.ts · stats.ts · socials.ts · blogCategories.ts
├── lib/
│   ├── schema.ts             # JSON-LD builders (Org, LocalBusiness, FAQ, Article…)
│   └── blog.ts               # reading time, TOC, date format, post queries
├── components/               # 20 reusable .astro components (see below)
├── layouts/Layout.astro      # base shell: SEO + Header + Footer + sticky CTA
├── pages/                    # routes (static + dynamic [slug] generators)
├── styles/global.css         # Tailwind v4 @theme tokens + component classes
└── entrypoints/alpine.js     # Alpine store (mobile menu, scroll state)
```

### Data-driven by design
Nothing is hardcoded in components. Add a service to `data/services.ts` and you
automatically get a mega-menu entry, a `/services/<slug>` page, schema markup and
footer links. The same pattern drives projects, locations and the blog.

## SEO system

`src/components/SEO.astro` is the single SEO surface. Every page passes its meta
through `Layout` → `SEO`, which renders:

- Title, meta description, keywords, canonical, robots
- Open Graph + Twitter card tags
- Geo tags (region / placename / position / ICBM)
- JSON-LD: **Organization** (always) + per-page **Breadcrumb**, **LocalBusiness**,
  **FAQPage**, **Service**, **Article** and **CreativeWork/Project** schema

Schema builders live in `src/lib/schema.ts`. The XML sitemap is generated by
`@astrojs/sitemap`; an HTML sitemap lives at `/sitemap`.

## Images
Real photography is a client deliverable. Until it's added, every `<img>` whose
file is missing renders a **branded gradient placeholder** (see `Layout.astro`).
The full image plan — filenames, alt, title, caption, description and keywords for
every slot — is documented in [`public/images/IMAGES.md`](public/images/IMAGES.md).
Folder structure: `public/images/{brand,home,services,projects,blog,team,locations}`.

## Performance & accessibility
- Static HTML, **near-zero JS** (only Alpine), prefetch on viewport
- Responsive `<img>` with explicit `width`/`height`, lazy-loading below the fold,
  `fetchpriority="high"` on heroes
- Fonts via `display=swap` + preconnect
- Skip-to-content link, focus-visible rings, semantic landmarks, `aria-*` on
  interactive widgets, `prefers-reduced-motion` honored

## Extending / CMS migration
Because all content lives in typed `data/*.ts` files and a Markdown content
collection, swapping in a headless CMS (Sanity, Contentful, Storyblok…) means
replacing the data-loading layer only — components and pages stay untouched.

### Wiring the forms
`ContactForm.astro` is pre-wired for Netlify Forms (`data-netlify`) and includes
a client-side success state. Point the form `action` / submit handler at your CRM,
serverless function, or form provider to go live.

### Adding content
- **A service** → add an entry to `src/data/services.ts`
- **A project** → add an entry to `src/data/projects.ts`
- **A location** → add an entry to `src/data/locations.ts`
- **A blog post** → drop a Markdown file in `src/content/blog/` (see existing files for the frontmatter shape)

Update `astro.config.mjs` `site` and `src/data/company.ts` with the real brand details before launch.
